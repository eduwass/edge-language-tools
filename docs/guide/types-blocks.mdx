---
title: The @types block
description: Declaring a template's interface.
---

A `@types` block is the template's function signature. It can be written as a comment (works everywhere, zero setup) or as a real tag (needs one line of runtime setup, but formats with Prettier and reads like the rest of the template). Both accept the same bodies: an object literal or any TypeScript type expression.

<Tabs>
  <Tab title="Comment form">
    <CodeGroup>
    ```edge profile.edge
    {{--
    @types {
      user: { name: string, email: string }
      posts: { title: string, body: string }[]
    }
    --}}
    ```
    </CodeGroup>

    Works with plain Edge, no runtime changes. This is the default — reach for it first.
  </Tab>
  <Tab title="Tag form">
    <CodeGroup>
    ```edge profile.edge
    @types()
      user: { name: string, email: string }
      posts: { title: string, body: string }[]
    @end
    ```
    </CodeGroup>

    Real Edge doesn't know the `@types` tag yet, so it needs to be registered once at startup — otherwise it renders its body as literal text. `@edge-language-tools/edge-plugin` supplies a no-op tag that consumes it silently:

    ```ts app/edge.ts
    import { Edge } from 'edge.js'
    import { registerTypesTag } from '@edge-language-tools/edge-plugin'

    const edge = Edge.create()
    registerTypesTag(edge)
    ```

    If both forms appear in the same template, the tag form wins — the comment form is skipped entirely.
  </Tab>
  <Tab title="Imported type expression">
    The body accepts any TypeScript type expression, so shared shapes live in real `.ts` files. Works in either form:

    <CodeGroup>
    ```edge dashboard.edge
    {{-- @types import('./shared/layout.types.ts').Props --}}
    ```
    ```edge dashboard.edge
    @types(import('./shared/layout.types.ts').Props)
    @end
    ```
    ```ts shared/layout.types.ts
    export interface Props {
      user: { name: string; avatarUrl: string }
      nav: { label: string; href: string }[]
    }
    ```
    </CodeGroup>

    Best once several templates share a shape — one interface, many templates.
  </Tab>
</Tabs>

Imports resolve through your project's `tsconfig.json`, so path aliases work — including framework models:

```edge
{{-- @types { user: import('#models/user').User } --}}
```

<Panel title="Why the tag form has an empty `@end` for a single-line type">
  Edge's tokenizer requires a block tag to always have a matching `@end`, whether or not its body is empty — `@types(expr)` with no closing tag isn't valid Edge grammar. The comment form doesn't have this restriction, so it stays the more compact option for a one-line import expression.
</Panel>

## What's inferred (never declared)

- `@let(total = items.length * 2)` — `total: number`
- `@each(item in items)` — `item` from the array's element type
- Narrowing inside `@if(user.subscription)` blocks
- Edge globals (`truncate`, `html.escape`, case helpers, ...) are pre-typed

## Cross-file checking

- `@component('components/card', { title })` — the props object is checked against the component's own `@types`; typos error at the caller
- Supercharged shorthands (`@formInput(...)`, `@modal(...)`) resolve through Edge's filename-to-tag rules and get the same checking
- `@include('partials/nav')` — the caller's state must satisfy the partial's declared types
- Dynamic names (`@include(someVar)`) degrade gracefully to unchecked

<Panel title="The block is real TypeScript — tooling included">
  Everything inside `@types` gets full TS tooling in the editor: autocomplete for type names and `import()` specifiers, and errors anchored inside the block (a bad import path is flagged on the path itself).
</Panel>
