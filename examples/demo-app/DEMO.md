# Demo walkthrough

A bare edge.js app (no AdonisJS) exercising the full feature surface: typed
props, `@each`/`@else`, globals, `@include`, a supercharged component tag
(`@userCard`) with a `@name`/`@desc` doc header, `@slot`, and one untyped
template for gradual adoption. Run everything from
`examples/demo-app/`.

```sh
bun install        # from repo root
bun run codegen     # writes edge-templates.d.ts
bun run check       # edge-check over templates/
bun run typecheck   # tsc over src/ + generated d.ts
bun run dev          # serves at http://localhost:4780
```

```sh

@userCard({ displayName: user.displayNaem, bio: user.bio })
                              ^^^^^^^^^^^

```

Exact caret position on the typo ‚Äî the same identifier squiggles red in the
editor via the language server, before you ever run the CLI.

## Act 3 ‚Äî break a render call site

In `src/server.ts`, change the home page's `posts` argument to a
`string[]` instead of `{ title, body }[]`:

```ts
edge.render('home', { user, posts: posts.map(p => p.title), activePage: 'home' })
```

```sh

  Overload 1 of 2, '(templatePath: "home", state: { user: { name: string; }; posts: { title: string; body: string; }[]; activePage: string; }): Promise<string>', gave the following error.
    Type 'string[]' is not assignable to type '{ title: string; body: string; }[]'.
      Type 'string' is not assignable to type '{ title: string; body: string; }'.
  Overload 2 of 2, '(templatePath: never, state?: Record<string, any> | undefined): Promise<string>', gave the following error.
    Argument of type '"home"' is not assignable to parameter of type 'never'.
```

`TypedEdge` (generated by `edge-codegen`) rejects the call at the TypeScript

## Act 4 ‚Äî component contract

Pass an undeclared `size` prop from `templates/profile.edge` to the
`@userCard` tag (which is `components/user_card.edge` ‚Äî Edge's supercharged
filename-to-tag mapping, `user_card` -> `@userCard`):

```edge
@userCard({ displayName: user.displayName, bio: user.bio, size: 'large' })
```

```sh
$ bun run check
$ edge-check templates
templates/profile.edge:12:59 - error TS2353: Object literal may only specify known properties, and 'size' does not exist in type '{ displayName: string; bio?: string | undefined; }'.
@userCard({ displayName: user.displayName, bio: user.bio, size: 'large' })
                                                          ^^^^

1 error across 4 checked templates (1 unchecked, 5 total)
```

The component's own `@types` block is its contract ‚Äî callers are checked
against it, same as any other typed template.

## Act 5 ‚Äî the editor knows your components

`components/user_card.edge` opens with a doc header:

```edge
{{--
@name User Card
@desc Displays a user's name and a truncated bio, with an actions slot.
  Rendered as the supercharged @userCard tag (file: user_card.edge).
@types {
  displayName: string
  bio?: string
}
--}}
```

In VS Code, Cursor, or Zed (with the extension installed):

- Type `@` in any template ‚Äî the popup lists every built-in tag (each with a
  one-line doc) plus `@userCard`, showing its name, description, and props.
- Hover `@userCard` in `profile.edge` ‚Äî same doc card, in place.
- Hover `@include` or the `'partials/nav'` path string ‚Äî built-in docs and
  the partial's doc header, respectively.
- Cmd/Ctrl-click `@userCard` or `'partials/nav'` ‚Äî jumps to the `.edge` file.
- Inside `@userCard({ ... })`, prop names autocomplete from the component's
  `@types`.

None of this needed registration anywhere ‚Äî the doc header lives in the same
comment as `@types`, and stock Edge ignores comments.

## Act 7 ‚Äî strict mode: make types required where it matters

Gradual is the default, but this app opts components into required types via
`package.json`:

```json
"edge": {
  "check": {
    "requireTypes": ["templates/components/**"]
  }
}
```

Pages and partials can still adopt at their own pace, but a component with no
`@types` block is now an error. Add an untyped component and run the check:

```sh
$ printf '<div>untyped widget</div>\n' > templates/components/untyped-widget.edge
$ bun run check
templates/components/untyped-widget.edge:1:1 - error edge-check: template requires a @types block (edge.check.requireTypes)

1 error across 4 checked templates (2 unchecked, 6 total)
error: script "check" exited with code 1
```

The same rule surfaces in the editor as a diagnostic on line 1. Use
`"severity": "warn"` while migrating (reported, but exit code stays 0), and
`"exclude"` globs for permanent carve-outs like `templates/legacy/**`.

## Gotchas hit while building this demo

- A prop literally named `name` used to collide with the ambient DOM `lib`'s
  global `declare var name: string` (`window.name`). Fixed since: virtual
  files are emitted as modules (`export {}`), so props shadow ambient globals
  cleanly ‚Äî `user: { name }`-shaped props are fine now.
- `@include`'s inline second argument (`@include('foo', { bar: 1 })`) is not
  type-checked against the object literal ‚Äî the checker always validates the
  *caller's own* `state` against the included template's `@types`, mirroring
  how Edge merges the whole parent scope into an include. The demo's
  `partials/nav.edge` needs `activePage` to be part of the including
  template's own `@types`, not passed as an inline literal.
- Component slot bodies use `$slots.<name>()` at runtime
  (`{{{ await $slots.actions() }}}`), not the `@!slot(...)` form used in
  `packages/core`'s fixtures (that form is fixture-only shorthand for
  checker testing, not real edge.js syntax). `$slots` and `$caller` weren't
  declared as implicit globals in `packages/core/src/globals.ts`; added both
  so component bodies referencing them type-check.
